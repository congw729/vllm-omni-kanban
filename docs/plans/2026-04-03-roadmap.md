# vLLM-Omni-Kanban Roadmap

**Date:** 2026-04-03
**Status:** Draft

## Summary

This roadmap focuses the next stage of vLLM-omni-kanban on two primary product surfaces:

1. Model performance and benchmarks
2. Test case information

The current codebase already provides a dashboard framework, model pages, and a naive Buildkite nightly artifact downloader. The roadmap therefore prioritizes evolving the existing implementation instead of following the older March plans literally.

## Current Baseline

- Buildkite nightly artifact download already exists and should be treated as the initial ingestion path.
- The dashboard framework and chart generation already exist in the repo.
- Qwen3 Omni performance support exists on a personal `optim` branch and should be merged into the main roadmap.
- Qwen Image already has a page shell, but its performance view still needs to be completed using the Buildkite-derived data pipeline.

## April Plan

### 1. Stabilize Buildkite Data Ingestion @congw729

- Productize the current Buildkite nightly fetch path.
- Normalize downloaded performance artifacts into the kanban result schema.
- Make the Buildkite source contract explicit:
  build selection, artifact selection, preserved metadata, and failure handling.

### 2. Merge and Ship Qwen3 Omni Performance @congw729

- Bring the `optim` branch implementation into the mainline roadmap.
- Confirm metric definitions and chart behavior match the raw Buildkite artifact fields.
- Treat this as the first production-ready important-model performance page.

### 3. Add Qwen Image Performance @congw729

- Support Qwen Image benchmark and performance extraction from Buildkite artifacts.
- Complete the Qwen Image performance view using the normalized data pipeline.
- Avoid one-off model-specific ingestion logic where possible.

### 4. Add Test Case Information v1 @yenuo26

- Scope the first milestone to suite-summary information rather than full per-case history.
- Show pass/fail counts, top failing suites or cases, and links back to Buildkite.
- Keep this as a summary-oriented dashboard feature rather than a full test analytics system.

## May Plan

### 1. Expand Model Coverage

- Add the next model-specific performance pages after Qwen3 Omni and Qwen Image.
- Recommended next order:
  Qwen Image Edit, then Qwen3 TTS, then WAN2.2.
- Generalize model metric mapping so future models require minimal code changes.

### 2. Improve Dashboard UX

- Improve navigation between homepage, model pages, and test-case views.
- Add comparison-friendly views where useful, especially cross-hardware and recent-regression summaries.
- Improve page readability and reporting flow before deeper analytics work.

## Key Decisions Already Made

- Qwen3 Omni should be merged in April.
- Test case information v1 means suite summary, not per-case historical storage.
- May should be sequenced as coverage first, UX second.
- The roadmap should evolve the current codebase and Buildkite prototype rather than restart from the March v1 plan.

## Details Still To Decide

### Buildkite Ingestion Contract

- Exactly which nightly build should be considered authoritative.
- Which artifact types are required for performance and test-case views.
- How much Buildkite provenance should be exposed in the UI.

Recommended default:
show build number, commit, run date, and a source link on relevant pages.

### Test Case Summary Source

- Whether the dashboard should read Buildkite HTML/statistics artifacts directly.
- Or whether ingestion should normalize test-case summary into JSON first.

Recommended default:
normalize into JSON during ingestion so the dashboard stays simple.

### Performance View Structure

- Whether benchmark and performance should remain a single combined model page.
- Or whether they should split into separate sections or tabs later.

Recommended default:
keep one page per important model with separate sections if needed.

### Post-April Model Priority

- Confirm the exact model order for May after Qwen3 Omni and Qwen Image.
- Revisit priority if artifact quality or team demand changes.

## Acceptance Signals

- A nightly Buildkite run can be resolved and its required artifacts downloaded reliably.
- Qwen3 Omni data appears in the mainline dashboard after merge.
- Qwen Image metrics are normalized and rendered from Buildkite artifacts.
- Test-case suite summary renders from Buildkite-derived data and links back to the source build.
- Adding the next model in May does not require a pipeline rewrite.
- UX improvements do not break the existing homepage or model pages.
