# vLLM-Omni Kanban — Product Requirements Document

**Date:** 2026-03-13
**Status:** Draft
**Author:** Engineering team

---

## 1. Problem Statement

The vLLM-omni project runs CI tests daily across 5 multimodal models and 5 hardware platforms. Currently, results are scattered across CI logs with no unified view. Engineers must manually check logs to detect regressions, and there is no automated alerting when stability, performance, or accuracy degrades. This causes delayed incident detection and wasted engineering time.

---

## 2. Goals

| Goal | Success Metric |
|------|----------------|
| Centralize CI results into one dashboard | All 5 models × 5 hardware results visible in one page |
| Enable regression detection | Alert sent within 1 hour of a regression being detected |
| Retain historical data for trend analysis | 90 days of rolling history queryable on the dashboard |
| Reduce manual monitoring effort | Zero manual log-checking required for routine status checks |
| Extensible to new models and hardware | Adding a new model requires only config change, no code change |

---

## 3. Non-Goals

- This is **not** a real-time streaming dashboard (daily batch updates are sufficient)
- This is **not** a CI/CD orchestration tool (it consumes CI results, does not run tests)
- This is **not** a general-purpose observability platform (scoped to vLLM-omni only)
- This does **not** support user authentication or access control (public GitHub Pages)
- This does **not** store raw model outputs or test artifacts (metrics only)

---

## 4. Users & Stakeholders

| Role | Need |
|------|------|
| **vLLM-omni engineers** | Know when a model broke on a specific hardware platform without checking CI logs |
| **Hardware platform owners** (NVIDIA, AMD, Ascend) | Verify their platform is stable across model updates |
| **Model owners** | Track accuracy and performance trends over time |
| **Engineering managers** | High-level stability snapshot for weekly reviews |

---

## 5. Functional Requirements

### 5.1 Data Ingestion

- **FR-01:** The system MUST accept CI result pushes from vLLM-omni CI via GitHub `repository_dispatch` event
- **FR-02:** Each CI result MUST include: timestamp, git commit SHA, hardware platform, model name, and at minimum `pass_rate` and `latency_p99_ms`
- **FR-03:** The system MUST reject and log malformed or schema-invalid results without crashing
- **FR-04:** The system MUST deduplicate results with identical `(timestamp, hardware, model)` tuples
- **FR-05:** Results older than 90 days MUST be pruned from storage on each update

### 5.2 Dashboard

- **FR-06:** The dashboard homepage MUST display a summary card showing overall pass rate, average latency, and latest commit date
- **FR-07:** The dashboard MUST show per-hardware status (A100, H100, H20, MI300X, Ascend) with current pass/fail state
- **FR-08:** The dashboard MUST show a model × hardware matrix with per-cell status indicator
- **FR-09:** The dashboard MUST include trend charts for 7-day and 30-day windows
- **FR-10:** Daily reports MUST be auto-generated as dated Markdown pages (e.g., `reports/2026-03-13.md`)

### 5.3 Alerting

- **FR-11:** The system MUST send an alert when pass rate drops below 80% (critical) or 90% (warning)
- **FR-12:** The system MUST send an alert when latency P99 exceeds 1000ms
- **FR-13:** The system MUST send an alert when crash count ≥ 3
- **FR-14:** The system MUST detect regressions vs 7-day baseline: pass rate −5%, latency P99 +20%, TTFT +20%, throughput −15%, benchmark score −5%
- **FR-15:** Alert notifications MUST be sent via WeChat Enterprise webhook and Email (SMTP)
- **FR-16:** The system MUST suppress duplicate alerts for the same metric breach on the same hardware/model combination for 24 hours (cooldown)
- **FR-17:** Alert history MUST be stored in `alerts.json` and displayed on the dashboard

### 5.4 Configuration & Extensibility

- **FR-18:** Models and hardware platforms MUST be defined in `config.json` — adding a new model requires no code changes
- **FR-19:** Per-model metric requirements (required vs optional) MUST be configurable
- **FR-20:** Alert thresholds MUST be configurable per model or globally in `config.json`

---

## 6. Non-Functional Requirements

| ID | Requirement |
|----|-------------|
| **NFR-01** | Dashboard MUST load in < 3 seconds on a standard browser |
| **NFR-02** | Data processing pipeline MUST complete within 5 minutes of CI result push |
| **NFR-03** | System MUST handle the case where < 7 days of baseline data exist (degrade gracefully, skip regression check) |
| **NFR-04** | Storage per day MUST not exceed 500KB of JSON data |
| **NFR-05** | Test coverage for alerting logic MUST be ≥ 95% |
| **NFR-06** | Test coverage overall MUST be ≥ 80% |
| **NFR-07** | The system MUST be deployable entirely via GitHub Actions with no external infrastructure |

---

## 7. Constraints

- **GitHub Pages** for hosting (free, no auth, public)
- **GitHub Actions** for processing (no external CI/CD)
- **Python 3.11** for scripts
- **MkDocs Material** for dashboard rendering
- **ECharts** for interactive charts (embedded via MkDocs macros plugin)
- Secrets (WeChat webhook, SMTP credentials) stored in GitHub Secrets only

---

## 8. Data Retention & Storage

- CI results stored as **per-date JSON files** (`data/results/YYYY-MM-DD.json`) for efficient reads and git diffs
- A lightweight `data/index.json` tracks the list of available dates and last-updated timestamp
- 90-day rolling window enforced on every write
- Daily Markdown reports stored in `docs/reports/` (not pruned — permanent archive)

---

## 9. Out-of-Scope for v1

- Multi-branch tracking (main branch only)
- Per-test-case drill-down (summary metrics only)
- Custom dashboard filtering by user
- Mobile-optimized layout
- Slack / DingTalk / PagerDuty notifications (WeChat + Email only for v1)
